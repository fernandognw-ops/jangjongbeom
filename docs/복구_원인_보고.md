# 대시보드 데이터 복구 원인 보고

## 1. 적용한 수정 사항

### 1) get_inbound_outbound_totals_by_channel 롤백
- **원인**: 새로 추가한 `get_inbound_outbound_totals_by_channel` RPC가 기존 테이블/스키마와 충돌
- **조치**: Summary API 사용 중단, `/api/inventory` 직접 테이블 참조 방식으로 롤백
- **변경**: InventoryContext가 Summary API 대신 `/api/inventory`만 호출하도록 수정

### 2) 재고 금액 0원 해결
- **원인**: 
  - Summary API 집계 로직에서 스냅샷·마스터 매칭 오류 가능성
  - `computeTotalValueFromSnapshot`에서 단가 0 시 마스터 단가 fallback 미적용
- **조치**:
  - `computeTotalValueFromSnapshot(snapshot, products)`에 products 인자 추가
  - 스냅샷 `unit_cost`가 0이거나 비정상이면 `inventory_products.unit_cost` 사용
  - 바코드(code) 기준으로 마스터 단가 매칭 후 `수량 × 단가` 합산

### 3) 품절 임박 228건 오표기 수정
- **원인**: 안전재고(safetyStock) 매칭 오류로 재고 > 0인 품목까지 품절로 분류
- **조치**: `getStockStatus`에서 **재고 > 0이면 품절 임박에서 제외**하도록 수정
  - `stock <= 0`일 때만 `"out"` (품절 임박)
  - `stock > 0`이면 low / overstock / normal 중 하나만 반환

### 4) Summary API 채널 로직 제거
- `get_inbound_outbound_totals_by_channel` 호출 제거
- 스냅샷 없을 때: `inventory_inbound`, `inventory_outbound` 직접 조회 후 재고 계산
- 채널별 재고: `stockByProductByChannel = { coupang: {}, general: { ...stockByProduct } }`로 단순화

## 2. DB 테이블 구조 (변경 없음)

| 테이블 | 용도 |
|--------|------|
| `inventory_products` | 품목 마스터 (code, unit_cost 등) |
| `inventory_current_products` | 현재 운영 품목 코드 목록 (product_code만, 수량 없음) |
| `inventory_stock_snapshot` | 재고 스냅샷 (product_code, quantity, unit_cost) |
| `inventory_inbound` | 입고 내역 |
| `inventory_outbound` | 출고 내역 |

**참고**: `inventory_current_products`에는 수량 컬럼이 없음. 재고 수량은 `inventory_stock_snapshot`에서만 사용.

---

## 3. 쿼리/로직 오류 정리

| 구분 | 내용 |
|------|------|
| **Summary API 우선 사용** | 집계 RPC 충돌로 잘못된 데이터 반환 → `/api/inventory` 직접 조회로 복구 |
| **채널 RPC 충돌** | `get_inbound_outbound_totals_by_channel` 제거, 직접 테이블 참조로 변경 |
| **재고 금액 계산** | 스냅샷 단가 0일 때 마스터 단가 fallback 추가 |
| **품절 판정** | 재고 > 0이면 품절 아님으로 하드코딩 |

---

## 4. 복구 후 확인 사항

1. 대시보드 새로고침 후 재고 금액이 0이 아닌지 확인
2. 품절 임박 건수가 실제 재고 0인 품목 수와 일치하는지 확인
3. 채널 탭(전체/쿠팡/일반) 동작 확인 (현재는 general만 사용)
