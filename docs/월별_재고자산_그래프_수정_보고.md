# 월별 실제 재고 자산금액 그래프 — 원인 및 수정 보고

## 1. 원인

| 항목 | 내용 |
|------|------|
| **직접 원인** | `inventory_stock_snapshot` 조회가 **`.limit(50000)` 단일 요청**이었고, **페이지네이션 없음**. |
| **현상** | PostgREST가 반환하는 5만 행이 **특정 snapshot_date(예: 26-03)에 치우치거나** 과거 스냅샷 행이 누락 → `maxDateByMonth`에 **한 달만** 채워짐. |
| **집계 로직** | 월별 마지막 `snapshot_date` 선택 로직 자체는 이미 구현되어 있었으나, **입력 데이터가 불완전**해 과거 월이 비어 보임. |

## 2. 수정 파일

| 파일 | 변경 |
|------|------|
| `src/app/api/category-trend/route.ts` | 재고 스냅샷을 `fetchAllRows`로 **`snapshot_date >= dateFrom`(차트 14개월 시작일)** 전량 로드. `limit(50000)` 제거. 14개월 슬롯(`months`)을 fetch **이전**에 계산해 조회 범위와 일치. |

## 3. 재고 자산 집계 규칙 (고정)

1. `inventory_stock_snapshot`에서 **`snapshot_date >= 차트 시작월 1일`**인 **모든 행**을 읽음 (페이지네이션).
2. `snapshot_date` → `YYYY-MM`(`monthKey`)별로 **그 월의 최대 일자** = 해당 월 **마지막 스냅샷일** 1개.
3. 그 일자에 해당하는 행만 남겨 **카테고리별** `total_price` 합산 (`total_price` 없으면 `qty × unit_cost` 등 기존 폴백).
4. 차트 월 축: **최근 14개월** (예: **25-02 ~ 26-03** 형태, 현재 달 기준 롤링).

**적재(웹)와의 정합:** DB에는 **달마다 스냅샷 일자가 1종만** 남도록 `commitProductionSheet`가 **같은 달 전체 DELETE 후 INSERT**한다. → 그래프의 「월별 마지막 스냅샷」과 맞춤. (`docs/재고스냅샷_운영규칙.md`)

## 4. 검증 SQL

`scripts/verify_snapshot_monthly_assets.sql`

- §1: 일자별 row 수·금액  
- §2: 월별 마지막 스냅샷일  
- §3: 그 일자 기준 카테고리별 합  

## 5. 응답 검증 (`GET /api/category-trend`)

| 확인 | 방법 |
|------|------|
| 과거 월 채움 | `monthlyValueByCategory["2025-02"]` 등에 마스터 카테고리 키가 **0이 아닌 값**이 있는지 |
| 26-03만 아님 | `Object.keys(monthlyValueByCategory).length` 및 여러 `YYYY-MM` 키 |
| 스택 막대 | `CategoryTrendChart` — `monthlyValueByCategory[month][cat]`가 월마다 존재 |

## 6. 수정 전/후 응답 비교 (요약)

| 항목 | 수정 전 | 수정 후 |
|------|---------|---------|
| `monthlyValueByCategory` | 대개 **현재 스냅샷이 있는 한 달**만 유효 금액 | **DB에 스냅샷이 있는 각 월**마다 마지막 일자 기준 금액 |
| 스냅샷 fetch | 최대 5만 행, 임의 순서 | `snapshot_date` 오름차순 페이지네이션 전량 |
| 서버 로그 | `snapshot=N` | `snapshot_rows=…`, `months_with_last_snapshot=…` |

배포 후 브라우저 Network 탭에서 `/api/category-trend` JSON의 `monthlyValueByCategory`를 확인하면 됩니다.
