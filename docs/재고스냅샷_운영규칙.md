# 재고 스냅샷(`inventory_stock_snapshot`) 운영 규칙

## 1. 원칙

| # | 규칙 |
|---|------|
| 1 | 재고 데이터는 **당월(달력 월) 기준**으로 관리한다. |
| 2 | **같은 달**에 새 파일이 업로드되면, 그 **달에 이미 쌓인 스냅샷은 모두 삭제**한 뒤 새 파일 내용으로 **INSERT**한다. |
| 3 | 한 달이 끝난 뒤 DB에는 그 달에 대해 **최종적으로 `snapshot_date`가 하나**만 남는 것이 목표다 (업로드 파일이 하루 기준이면 그 날짜 한 종류). |
| 4 | **월별 재고 자산 그래프**(`GET /api/category-trend`의 `monthlyValueByCategory`)는 각 달 **마지막 `snapshot_date` 1일**의 행만 합산한다. |

## 2. 구현 (웹 DB 반영)

- 파일: `src/lib/commitProductionSheet.ts`
- 동작: 업로드 파싱 결과의 `snapshot_date`에서 **서로 다른 `YYYY-MM`**을 구한 뒤, 각 월에 대해  
  `snapshot_date >= YYYY-MM-01` **이고** `snapshot_date < 다음달 1일` 인 행을 **전부 DELETE**, 그다음 파일 행을 **INSERT**.
- **같은 달 재업로드**: 예전에 3/1·3/15가 DB에 있어도, 새 파일이 `2026-03-25`만 포함하면 **3월 전체**가 비워진 뒤 3/25만 들어간다.

## 3. 구현 (로컬 `integrated_sync.py --apply`, 비권장)

- 파일: `scripts/integrated_sync.py` — `delete_stock_snapshot_calendar_months`
- 웹과 동일하게 **달력 월 단위** 구간 삭제 후 insert.

## 4. 입고·출고와의 차이

- 입고·출고: 파일에 나온 **일자 목록**만 DELETE 후 INSERT (일 단위 교체).
- 재고 스냅샷: 파일에 나온 **월 목록**마다 **그 달 전체** DELETE 후 INSERT (월 단위 교체).

## 5. 검증 SQL

- `scripts/verify_snapshot_monthly_assets.sql`
  - §1~§3: 일자·월별 마지막 스냅샷·카테고리 합 (그래프와 맞춤)
  - **§4**: `HAVING COUNT(DISTINCT snapshot_date::date) > 1` — **한 달에 스냅샷 일자가 2종 이상**이면 운영 규칙 위반·미정리 상태로 점검
