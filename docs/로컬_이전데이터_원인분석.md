# 로컬 화면 이전 데이터 원인 분석

**분석 일시**: 2025-03-17  
**대상**: 로컬 개발환경 (localhost:3007)  
**배포**: 정상화됨 (0건 표시)

---

## 1. localStorage inventory-* 키 화면 영향

### 영향 조건

| useSupabaseInventory | 데이터 출처 | localStorage 영향 |
|----------------------|-------------|-------------------|
| **true** | API 응답 (supabaseDerived) | **없음** |
| **false** | baseStock, transactions, products (localDerived) | **있음** – storage.load*()로 로드 |

### 현재 로컬 상황

로컬 API가 **326건**을 반환하므로 `useSupabaseInventory = true`가 됩니다.  
이 경우 **localStorage는 사용되지 않고**, 화면은 전부 API 응답만 사용합니다.

**결론**: 로컬에서 이전 데이터가 보이는 원인은 **localStorage가 아님**.

---

## 2. empty_data / fetch_error 시 clearLocalInventoryData() 실행 여부

### 코드 흐름

- **empty_data**: `items.length === 0 && totalVal === 0`일 때만 진입
- **fetch_error**: API 예외 또는 `!snapshotRes?.ok`일 때 catch 블록 진입

### 현재 로컬

- localhost `/api/inventory/quick`가 **326건**을 반환
- `items.length > 0`, `totalVal > 0` → **empty_data 분기 미진입**
- API 정상 응답 → **catch 블록 미진입**

**결론**: 로컬에서는 `clearLocalInventoryData()`가 **호출되지 않음** (empty_data/fetch_error가 발생하지 않음).

---

## 3. 로컬 dev 서버 최신 코드 실행 여부

### 확인 사항

- `SUPABASE_SINGLE_SOURCE = true`는 `InventoryContext.tsx`에 있음
- empty_data/fetch_error 시 `clearLocalInventoryData()` 호출 로직도 코드에 포함됨

### 실행 여부

- 로컬 API가 **데이터를 반환**하므로 empty_data/fetch_error 경로가 실행되지 않음
- 따라서 `SUPABASE_SINGLE_SOURCE` 관련 정리 로직이 **실행될 기회가 없음**

**결론**: 최신 코드가 적용되어 있어도, 현재 로컬 응답 특성상 해당 정리 로직은 실행되지 않음.

---

### 실제 측정 결과

```

→ productCount: 326
→ totalValue: 789,412,321

→ _supabase_project_ref: (응답에 포함 시 slnmhblsxzjgmaqbfbwa 예상)
```

### 배포와 비교

| 구분 | productCount | totalValue |
|------|--------------|------------|
| **로컬** (localhost:3007) | **326** | **789,412,321** |
| **배포** (jangjongbeom.vercel.app) | 0 | 0 |

**결론**: 로컬 API는 **0건이 아니라 326건**을 반환하고 있음.

## 5. API는 0건인데 화면만 데이터가 있는 경우 (해당 없음)

현재 로컬은 **API가 0건이 아님**.  
따라서 “API는 0건인데 화면만 데이터가 있다”는 상황은 **해당하지 않음**.

---

## 6. 실제 원인

### 핵심 원인: 로컬 API가 데이터를 반환함

로컬 dev 서버의 `/api/inventory/quick`가 **326건**을 반환하고 있어,  
화면은 이 API 응답을 그대로 표시하고 있습니다.

### 가능한 이유

| 가능성 | 설명 |
|--------|------|
| **1) Supabase 프로젝트 불일치** | 로컬 `.env.local`이 배포(Vercel)와 **다른 Supabase 프로젝트**를 가리키고, 그 프로젝트에 데이터가 남아 있음 |
| **2) 동일 프로젝트, 데이터 존재** | 같은 Supabase를 쓰는데, 로컬에서만 데이터가 보이는 경우는 거의 없음. 단, 로컬에서 별도 적재를 했다면 가능 |
| **3) dev 서버 재시작 전 env** | `.env.local`을 수정한 뒤 dev 서버를 재시작하지 않아, **이전 Supabase URL/키**를 계속 사용 중 |

### 확인 방법

1. **로컬 API 응답의 project_ref 확인**
   ```text
   GET http://localhost:3007/api/inventory/quick
   → 응답 JSON의 _supabase_project_ref 값 확인
   ```

2. **로컬 .env.local과 배포 환경변수 비교**
   - `NEXT_PUBLIC_SUPABASE_URL`
   - `NEXT_PUBLIC_SUPABASE_ANON_KEY`
   - 배포와 동일한 Supabase를 바라보는지 확인

3. **Supabase 대시보드에서 프로젝트 row count 확인**
   - `inventory_stock_snapshot` row 수
   - 로컬 `.env.local`에 설정된 프로젝트와 일치하는지 확인

---

## 7. 정리 절차 (로컬 0건으로 맞추기)

1. **로컬 .env.local**이 **배포와 동일한 Supabase 프로젝트**를 가리키는지 확인
2. **dev 서버 재시작**: `.env.local` 수정 후 `npm run dev` 재실행
3. **브라우저 강력 새로고침**: Ctrl+Shift+R
4. **API 재확인**: `http://localhost:3007/api/inventory/quick`가 0건을 반환하는지 확인

## 8. 요약

| 항목 | 결과 |
|------|------|
| localStorage 영향 | **없음** (useSupabaseInventory=true이므로 API만 사용) |
| clearLocalInventoryData() 실행 | **미실행** (empty_data/fetch_error 미발생) |
| 최신 코드 적용 | **적용됨** (단, 정리 로직이 실행될 조건이 아님) |
| localhost quick API | **326건 반환** (0건 아님) |
| 초기 state에 이전 데이터 | **해당 없음** (API가 데이터를 반환하고 있음) |

**실제 원인**: 로컬 dev 서버가 사용하는 Supabase에 데이터가 있어, `/api/inventory/quick`가 326건을 반환하고, 화면이 이를 그대로 표시하고 있음.
