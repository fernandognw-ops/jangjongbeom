# 적재 로직 수정 및 잘못된 데이터 정리 보고

## 1. 원인 요약

| 현상 | 원인 |
|------|------|
| 재고 스냅샷이 특정일(예: 2026-03-18)만 존재, 과거 자산 그래프 비정상 | `commitProductionSheet`가 **당월 `snapshot_date`만** DB에 반영. 과거월 행은 버려짐. |
| 판매/출고·카테고리 수량 왜곡 | **웹 커밋**은 입·출고에 대해 원칙적으로 날짜별 DELETE 후 INSERT였으나, **Python `integrated_sync.py`**는 **당월 전체 구간만** 삭제·필터해 **파일에 없는 과거일**과 섞이거나, **같은 월 재실행 시 upsert/당월 삭제**로 기대와 다른 누적이 발생할 수 있음. |
| 기준일 불일치 | 재고 기준일 셀 공란 시 **업로드 당일**로 채워져 파일 의미와 어긋날 수 있음. |

## 3. 삭제·재적재 범위 (데이터 정리 방안)

### 3.1 `inventory_stock_snapshot`

- **현황:** 과거 스냅샷이 없거나 한 날짜만 있으면 시계열·자산이 의미 없음.
- **제안:** 검증된 **전 기간 엑셀**을 보유했다면  
  - **옵션 A:** `TRUNCATE inventory_stock_snapshot;` (또는 `DELETE` 전체) 후 **날짜 오름차순**으로 파일 재업로드.  
  - **옵션 B:** 잘못된 날짜만 `DELETE WHERE snapshot_date = '…'` 후 해당 파일만 재반영.
- **주의:** `2026-03-18` 등 **검증 완료 일자**는 재업로드 시 **같은 파일**로 덮어쓰면 유지; 다른 실험 파일로 덮어쓰면 덮어쓰기됨.

### 3.2 `inventory_outbound` / `inventory_inbound`

- **현황:** 동일 `outbound_date`/`inbound_date`에 행이 **중복 누적**되었을 가능성.
- **제안:**  
  1. `scripts/verify_inventory_load.sql`의 **§5·§6**으로 중복 행 존재 여부 확인.  
  2. **신뢰할 수 있는 원본 엑셀**이 있는 **날짜 범위**를 정한 뒤, 예시:  
     ```sql
     DELETE FROM inventory_outbound
     WHERE outbound_date >= '2025-01-01' AND outbound_date < '2026-04-01';
     ```  
     입고도 동일하게 `inbound_date` 기준.  
  3. 해당 기간 파일을 **과거→최신 순**으로 다시 업로드(웹 승인).  
- **주의:** 출고 테이블은 **PK가 `id`(UUID)**인 경우가 많아, **DELETE 없이 insert만 반복**하면 중복이 쌓임. 이번 수정 후 웹 경로는 **파일에 있는 날짜만 먼저 DELETE**하므로 동일 파일 재실행 시 **해당 일 전체 교체**로 정리됨.

## 4. 재적재 순서 (권장)

1. **백업** (Supabase 백업 또는 테이블 export).  
2. **정리 DELETE** (위 §3 범위 확정 후).  
3. **원본 엑셀**을 **기준일 오름차순**으로 정렬한 뒤 순차 업로드.  
4. 각 파일 후 `scripts/verify_inventory_load.sql` §1–§4 실행.  
5. 대시보드·`quick`/`snapshot` API로 **최종일·월합** 샘플 검증.

**동작 요약 (코드 반영 후):**

- **재고:** 업로드에 등장한 **`snapshot_date` 값마다** 기존 행 삭제 후 insert (같은 날 재업로드 = 그 날만 교체).  
- **입고·출고:** 업로드에 등장한 **`inbound_date` / `outbound_date` 집합**만 삭제 후 insert (**append 누적 아님**).  
- **기준일 공란:** 파일명에 `YYYY-MM-DD` 또는 `YYYYMMDD`가 있으면 그날을 스냅샷 기본일로 사용.
