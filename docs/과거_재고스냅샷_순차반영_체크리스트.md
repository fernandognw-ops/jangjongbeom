# 과거 재고 스냅샷 순차 반영 — 운영 체크리스트

> **목표:** `2026-03-18` 등 최신 정상 데이터를 건드리지 않고, 과거 엑셀을 **날짜 순**으로 안전히 누적한다.  
> **DB:** `inventory_stock_snapshot`  
> **PK:** `(product_code, dest_warehouse, storage_center, snapshot_date)`

---

## 0. 반드시 읽을 것 (시스템 제약) — 2026-03 갱신

| 구분 | 내용 |
|------|------|
| **웹 생산수불 승인 적재** (`commitProductionSheet`) | 재고·입고·출고 모두 **파일에 나온 날짜**만 대상: 해당 **`snapshot_date` / `inbound_date` / `outbound_date` 집합**을 먼저 DELETE 후 INSERT. **과거월 재고도 반영됨.** |
| **Python `integrated_sync.py` (`--apply`)** | 웹과 동일하게 **파일에 포함된 날짜만** 삭제 후 반영 (당월 제한 제거). |

**기준일 셀이 비어 있으면** 파일명에 `YYYY-MM-DD` 또는 `YYYYMMDD`가 있으면 그 날을 스냅샷 기본일로 사용 (`defaultDateFromFilename`).

---

## 1. 원칙 요약 (운영 합의)

| # | 원칙 | 운영 메모 |
|---|------|-----------|
| 1 | **가장 오래된 파일부터** 순서대로 | 파일명/메타보다 **`snapshot_date`(재고 기준일)** 기준으로 정렬해 순서 확정 |
| 2 | **파일별 `snapshot_date` 유지** | 엑셀 「기준일자」등 → 파싱된 `snapshot_date`가 DB에 그대로 가야 함 |
| 3 | **같은 날짜 재업로드 시에만 그 날짜 교체** | 아래 「동일 날짜 중복 방지」참고 |
| 4 | **`dest_warehouse` = 판매채널(쿠팡/일반), `storage_center` = 보관센터** | 엑셀 「판매 채널」「보관 센터」열 확인 (`rules.ts` 동의어) |
| 5 | **최신 정상일(예: 2026-03-18) 보호** | 해당 `snapshot_date`가 포함된 파일은 **마지막에** 두거나, 과거만 먼저 넣고 **해당 날짜는 건너뛰기** |

---

## 2. 파일 반영 순서 제안

1. **스냅샷 일자 목록 작성**  
   - 각 파일에서 재고 시트 기준일(또는 파일이 하루만 담당하면 그 날짜)을 적는다.  
2. **오름차순 정렬**  
   - 예: `2025-01-05` → `2025-01-12` → … → `2026-03-10` → …  
3. **최신 보호**  
   - `2026-03-18`이 이미 검증됐다면, 순차 목록에서 **해당 날짜는 제외**하거나 **전체 과거 반영 완료 후** 필요 시에만 단독 재검증.

**한 파일에 여러 `snapshot_date`가 섞여 있으면** (드물지만) 파일을 나누거나, 월/일 단위로 잘라서 적재하는 편이 검증에 유리하다.

## 4. 날짜별 적재 · 검증 절차 (한 파일/한 배치 단위)

### 4.1 적재 전

- [ ] 해당 배치의 **`snapshot_date` 목록** 확정 (하루인지, 여러 날인지).
- [ ] **최신 정상일(2026-03-18)이 이 배치에 섞이지 않았는지** 확인 (원칙 5).
- [ ] 엑셀: **판매 채널** / **보관 센터** 열 존재 및 헤더 동의어 확인 (`stock_sales_channel`, `storage_center`).

### 4.2 적재

- [ ] 웹 승인 또는 `integrated_sync.py --apply` 실행.

### 4.3 적재 직후 검증 (SQL)

아래 **§5** 실행.  
- 행 수·금액 합계가 기대와 비슷한지.  
- `dest_warehouse`가 `쿠팡`/`일반`만 있는지 (오타·공백).

### 4.4 이상 시

- [ ] **같은 날짜만** 롤백: `DELETE FROM inventory_stock_snapshot WHERE snapshot_date = 'YYYY-MM-DD';` 후 수정 파일로 재적재.  
- [ ] 최신일을 건드렸다면: 백업/PITR 절차(사내 문서) 준수.

## 5. 반영 후 확인 SQL (Supabase SQL Editor)

**5.1 날짜별 건수·수량·금액**

```sql
SELECT
  snapshot_date,
  COUNT(*) AS row_count,
  SUM(quantity) AS qty_sum,
  SUM(COALESCE(total_price, 0)) AS amount_sum
FROM inventory_stock_snapshot
WHERE snapshot_date BETWEEN '2025-01-01' AND '2026-12-31'
GROUP BY snapshot_date
ORDER BY snapshot_date;
```

**5.2 특정 일자 — 채널·보관센터 분포**

```sql
SELECT
  dest_warehouse,
  storage_center,
  COUNT(*) AS rows,
  SUM(quantity) AS qty
FROM inventory_stock_snapshot
WHERE snapshot_date = '2026-03-18'  -- 확인할 날짜로 변경
GROUP BY dest_warehouse, storage_center
ORDER BY dest_warehouse, storage_center;
```

**5.3 최신 정상일 존재 여부 (훼손 방지 확인)**

```sql
SELECT snapshot_date, COUNT(*) AS n
FROM inventory_stock_snapshot
WHERE snapshot_date = '2026-03-18'
GROUP BY snapshot_date;
```

**5.4 전체 기간 `dest_warehouse` 이상값 점검**

```sql
SELECT dest_warehouse, COUNT(*)
FROM inventory_stock_snapshot
GROUP BY dest_warehouse
ORDER BY COUNT(*) DESC;
```

(기대: 주로 `쿠팡`, `일반` — 레거시 문자열이 있으면 집계와 불일치 가능.)

## 6. 재고 스냅샷 교체 방식 (시스템 동작)

| 레벨 | 동작 |
|------|------|
| **DB 제약** | `PRIMARY KEY (product_code, dest_warehouse, storage_center, snapshot_date)` — 동일 키로 **두 번 INSERT 불가**. |
| **웹 커밋 / integrated_sync** | 업로드에 포함된 **달력 월(`YYYY-MM`)마다** `[해당월 1일, 다음달 1일)` 구간의 기존 행을 **전부 DELETE** 후 **INSERT**. |
| **운영 의미** | **같은 달**에 새 파일이 오면 그 **달의 예전 스냅샷 일자는 모두 사라지고** 새 파일의 `snapshot_date`만 남는다. 한 달에 **최종적으로 스냅샷 일자 1종**이 되도록 설계. |
| **다른 달** | 파일에 없는 달은 **DELETE 대상이 아님** (월 단위 격리). |

상세: `docs/재고스냅샷_운영규칙.md`
